---
description: Workflow for code refactoring to ensure consistency and documentation
---

#### !!최상위 중요 지침!!
1. **사전 설명 필수**: 어떤 도구(Tool)를 사용하거나 작업을 수행하기 전에, **반드시 사용자에게 무엇을 할 것인지 명확히 설명**해야 합니다. (설명 없이 바로 행동에 옮기지 마십시오.)
2. **실행 금지**: 분석이나 테스트를 시작하는 명령어(`npm run analyze`, `npm test` 등)를 직접 실행(`run_command`)하지 마십시오. 가이드만 제시하십시오.

이 워크플로우는 코드 리팩토링 작업 시 일관성과 문서화를 보장하기 위해 사용합니다.

1. **분석 (Analyze)**: 대상 파일을 분석하여 개선이 필요한 부분(예: 고정지연(static wait), 취약한 셀렉터(weak selector), 모달 동기화 누락 등)을 식별합니다.

2. **계획 및 문서화 (Plan & Document)**: **!!중요!!** `docs/PROJECT_BRIEFING.md` 파일에 "Refactoring Plan" 항목을 추가합니다.
   - **중요**: `implementation_plan.md` 아티팩트의 핵심 내용(목표, 변경 사항)을 요약하여 브리핑 문서에 반드시 포함시켜야 합니다.
   - 작성 형식:
     ```markdown
     ### [YYYY-MM-DD] Refactoring Plan: [기능/파일명]
     - **Goal**: ...
     - **Scope**:
       - [ ] 작업 항목
     ```

3. **안전 장치 (Safety First)**: **대규모 구조 변경 시** 아래 프롬프트 패턴을 참고하여 요청하거나 작업 전 스스로 점검합니다.
   - **기능 목록 사수**: "리팩토링하기 전에, 현재 파일에 구현된 주요 기능과 로직들을 먼저 리스트업하고, 이식 여부를 체크해."
   - **재작성 금지**: "코드를 새로 짜지 말고, 기존 코드를 최대한 유지하면서 아키텍처만 변경해. 함수 단위로 쪼개서 이식해."
   - **검증 기준**: "작업 완료 후, 기존 주요 기능이 새 코드 어디에 구현되었는지 매핑해서 보고해."

4. **승인 (Approval)**: 작성한 계획에 대해 사용자 검토(`notify_user`)를 요청하여 승인을 받습니다. **사용자가 채팅창에 '승인' 또는 '실행'이라고 입력하기 전까지는 절대 write_to_file이나 replace_file_content 같은 수정 도구를 호출하지 마라. 계획만 세우고 대기하라.**

5. **실행 (Execute)**: 코드를 변경합니다. 재작성(Rewrite)보다는 점진적 개선(Refactor)을 지향합니다.

6. **검증 (Verify)**: 안정성 검증을 위한 테스트 실행 명령어를 제공합니다. **어떤 경우에도 npm run analyze, npm test 등 분석이나 테스트를 시작하는 명령어를 직접 실행(run_command)하지 마라. 특히 '실행해볼게' 또는 '지워줘' 같은 요청을 받았을 때, 요청받은 '지우기' 작업까지만 수행하고 그 이후의 실행 프로세스는 반드시 사용자가 직접 터미널에서 입력할 수 있도록 가이드만 제시하라. 이를 위반할 경우 규칙 미준수로 간주한다.** 실행 여부를 묻지 말고, 사용자가 복사해서 바로 실행할 수 있도록 명령어만 코드 블록으로 제시하세요.

7. **문서 현행화 (Update Docs)**: 
   - `docs/PROJECT_BRIEFING.md`에 계획된 항목을 완료(check) 처리하거나, 새로 발견된 패턴이 있다면 "Lesson Learned"에 추가합니다.
   - **필수**: `docs/progress_log.md`에 분석/테스트 실행 결과("Run History")를 반드시 추가합니다. (성공/실패 여부, 발견된 페이지 수, 주요 변경점 포함)
